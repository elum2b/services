# Tasks service

Документ описывает поля заданий, которые отдает сервис `tasks`, и ожидаемую реакцию клиента/контроллера на каждое поле.

## Основные методы

### `task.list`

Возвращает список доступных обычных заданий, сгруппированный по группам:

```json
[
  {
    "key": "daily",
    "title": "Ежедневные задания",
    "description": "Ежедневные одиночные задания.",
    "tasks": []
  }
]
```

Параметры:

| Поле | Что передаем | Как использовать |
| --- | --- | --- |
| `locale` | Ключ локали, например `ru` или `en`. | Используется для выбора `title` и `description`. Если локализация не найдена, сервис вернет fallback. |
| `group_key` | Необязательный ключ группы. | Если нужно показать только одну группу заданий, передавать ее ключ. Если пусто, сервис вернет все доступные группы. |

### `task.partnerList`

Возвращает задания внешнего партнера в том же формате `Task`, что и обычные задания. Сервис скрывает детали партнера, создает/переиспользует внутренний `partner_issue`, а фронт работает с результатом как с обычным заданием.

Параметры:

| Поле | Что передаем | Как использовать |
| --- | --- | --- |
| `provider` | Код партнера: `tgrass`, `flyer`, `subgram`, `getbonus`. | Выбирает партнерский runtime/script. |
| `group_key` | Группа партнерских заданий, например `daily`. | Должна совпадать с настройкой `task_partner_config`. |
| `platform` | Платформа настройки, например `telegram`. | Должна совпадать с настройкой партнера в workspace. |
| `locale` | Локаль. | Передается партнеру и используется для публичного текста, если партнер его поддерживает. |
| `limit` | Максимум заданий от партнера. | Использовать для ограничения выдачи на экране. |
| `variables` | Публичные переменные для партнера. | Передавать только то, что не является секретом: `username`, `lang`, `chat_id` и подобные значения. |

Если партнер не настроен или отключен, метод может вернуть пустой список. Это не ошибка для UI: просто не показываем этот блок.

### `task.check`

Проверяет выполнение обычного/integration задания. Используется для подписок, бустов, внешних проверок и внутренних заданий, где выполнение надо подтвердить до `claim`.

Клиентская реакция:

| Результат | Что делать |
| --- | --- |
| `ready` | Обновить задание, показать кнопку получения награды. |
| `not_completed` или аналогичный статус | Оставить кнопку проверки/выполнения активной. |
| `not_found` | Убрать локальную ссылку на задание или обновить список. |
| Ошибка API | Показать ошибку и не менять локальный прогресс. |

### `task.partnerCheck`

Проверяет выполнение партнерского задания по `issue_ref`.

Параметры:

| Поле | Что передаем | Как использовать |
| --- | --- | --- |
| `issue_ref` | Значение `task.key` партнерского задания, обычно `partner_issue:<id>`. | Передавать ровно то, что пришло в задании. |
| `variables` | Дополнительные публичные значения. | Использовать только если конкретный партнер требует дополнительные данные для проверки. |

Клиентская реакция такая же, как у `task.check`: если `completed=true` или `status=ready`, обновить задачу и дать возможность забрать награду обычным `task.claim`.

### `task.start`

Используется для заданий, у которых `start_mode=required`. Сейчас это в первую очередь партнерские задания вроде GetBonus, где перед проверкой надо сгенерировать клик/ссылку.

Клиентская реакция:

| Поле ответа | Что делать |
| --- | --- |
| `started=true` | Считать задание начатым. |
| `action_url` | Открыть ссылку пользователю. |
| `task` | Заменить локальную версию задания на обновленную. |
| `status=not_found` | Обновить список заданий. |

Если `start_mode=none`, вызывать `start` не нужно.

### `task.claim`

Выдает награду за готовое задание. Для обычных, комплексных и партнерских заданий получение награды должно быть одинаковым.

Параметры:

| Поле | Что передаем | Как использовать |
| --- | --- | --- |
| `id` или `task_ref` | Идентификатор задания. | Для обычных заданий можно передавать `id`/`key`, для партнерских - `partner_issue:<id>`. Конкретный REST/socket слой может назвать поле по-своему, но в сервисе это `TaskRef`. |
| `operation_id` | Уникальный id операции в пределах workspace. | Обязателен для идемпотентности и не может повторно использоваться для другого обычного или партнерского задания. |

Клиентская реакция:

| Статус | Что делать |
| --- | --- |
| `claimed` | Показать полученную награду, обновить список/карточку. |
| `already_claimed` | Считать награду уже полученной, обновить список. |
| `not_ready` | Награда еще недоступна, оставить кнопку claim выключенной. |
| `not_started` | Сначала вызвать `task.start`. |
| `not_found` | Обновить список. |

## Модель группы

`task.list` возвращает массив групп.

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `key` | Стабильный ключ группы: `daily`, `complex`, `partner` и т.д. | Использовать как id вкладки/секции и как параметр `group_key`. |
| `title` | Локализованный заголовок группы. | Показывать пользователю. |
| `description` | Локализованное описание группы. | Показывать как подзаголовок, если дизайн это поддерживает. |
| `tasks` | Массив заданий. | Отрисовывать в порядке, в котором пришли. |

## Модель задания

```json
{
  "id": 1,
  "key": "daily.send_message",
  "group_key": "daily",
  "task_kind": "internal",
  "action_key": "message.send",
  "action_kind": "app_action",
  "claim_mode": "manual",
  "start_mode": "none",
  "target_count": 1,
  "payload": {},
  "image_url": null,
  "title": "Send a message",
  "description": "Send a message in the app.",
  "rewards": [],
  "progress": null,
  "conditions": []
}
```

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `id` | Числовой id задания внутри workspace. | Можно использовать как технический id для обычных заданий. Не использовать как публичный стабильный ключ между серверами. |
| `key` | Стабильный ключ задания. Для партнерских заданий может быть `partner_issue:<id>`. | Использовать как основной `task_ref` для check/claim/start, если API слой принимает строковую ссылку. |
| `group_key` | Ключ группы. | Использовать для группировки, фильтрации и аналитики UI. |
| `task_kind` | Тип задания на уровне бизнес-смысла. | Определяет общий сценарий UI. Значения описаны ниже. |
| `action_key` | Событие/действие, которое двигает прогресс. | Бэкенд должен записывать событие с таким ключом, когда действие реально произошло. Фронт обычно не интерпретирует это поле, кроме отладки. |
| `action_kind` | Класс действия: `app_action`, `amount_action`, `channel_subscribe`, `channel_boost`, `advertisement_view`, `external`, `composite`. | Помогает выбрать кнопку/сценарий проверки. |
| `claim_mode` | Как забирать награду: `manual` либо `auto` для внутренних задач. | При `manual` показывать кнопку claim только когда статус готов. `auto` выдаёт награду вместе с внутренним application event и не поддерживается integration/complex/partner task kinds. |
| `start_mode` | Нужно ли сначала начать задание: `none` или `required`. | Если `required`, перед check/claim надо вызвать start и обработать `action_url`. |
| `target_count` | Сколько единиц прогресса нужно. | Показывать прогресс `progress.progress / target_count`. |
| `payload` | Публичные данные для фронта. | Показывать ссылку, канал, кнопку, внешний URL, название партнера и т.д. Секретов здесь быть не должно. |
| `image_url` | Изображение задания. | Если есть, показать картинку/иконку. Если нет, использовать дефолтную иконку по `task_kind`/`action_kind`. |
| `title` | Локализованный заголовок. | Основной текст карточки. |
| `description` | Локализованное описание. | Дополнительный текст карточки. |
| `rewards` | Список наград. | Показывать пользователю до выполнения и после claim. |
| `progress` | Текущий прогресс пользователя. | Если отсутствует, считать прогресс нулевым и статус открытым. |
| `conditions` | Вложенные обязательные условия комплексного задания. | Показывать внутри карточки комплексного задания. Необязательные связи не увеличивают `target_count` родителя и остаются самостоятельными задачами. |

## `task_kind`

| Значение | Что означает | Как реагировать |
| --- | --- | --- |
| `internal` | Внутреннее задание приложения. | Действие выполняется в приложении. После действия бэкенд должен записать событие по `action_key`; фронт может вызвать `task.check` или обновить список. |
| `channel_subscribe` | Подписка на канал/чат. | По `payload` показать канал/ссылку. После действия пользователя вызвать `task.check`. |
| `channel_boost` | Буст канала. | По `payload` показать канал. После буста вызвать `task.check`. |
| `external_check` | Внешняя проверка через настроенный HTTP/integration provider. | Показать публичный payload, затем вызвать `task.check`; сервис сам спросит внешний API. |
| `external_confirming` | Внешнее подтверждение, где другая сторона может спросить нас или прислать событие. | UI работает как с обычным заданием, но готовность может прийти через callback/webhook. |
| `partner` | Партнерское задание, созданное через `task.partnerList`. | Работать через `task.partnerCheck` и обычный `task.claim`. |
| `complex` | Комплексное задание с несколькими условиями. | Показывать общий прогресс по `conditions`; claim доступен только когда все обязательные условия готовы. |

## `action_kind`

| Значение | Что означает | Как реагировать |
| --- | --- | --- |
| `app_action` | Обычное действие внутри приложения. | После выполнения действия записать событие на бэкенде. Например `pvp.play`, `referral.invite`, `task.claimed`. |
| `amount_action` | Действие с суммой. | Прогресс увеличивается на сумму/количество. Например пополнение от 100 звезд. |
| `advertisement_view` | Просмотр рекламы. | После успешного просмотра рекламы записать событие. |
| `channel_subscribe` | Проверка подписки. | Открыть канал, потом вызвать check. |
| `channel_boost` | Проверка буста. | Открыть канал/экран буста, потом вызвать check. |
| `external` | Внешнее действие/проверка. | Открыть внешний payload/action URL и использовать check/webhook. |
| `composite` | Родитель комплексного задания. | Не записывать напрямую как обычное событие; статус рассчитывается из `conditions`. |

## `claim_mode`

| Значение | Что делать |
| --- | --- |
| `manual` | Пользователь должен нажать кнопку получения награды. Фронт вызывает `task.claim`. |
| `auto` | Награда может быть выдана автоматически. Если такой режим появится в выдаче, фронт не должен требовать отдельную кнопку claim. |

## `start_mode`

| Значение | Что делать |
| --- | --- |
| `none` | Задание можно выполнять сразу. |
| `required` | Сначала вызвать start. Пока start не выполнен, check/claim могут вернуть `not_started`. |

## Прогресс

```json
{
  "progress": 3,
  "status": "open",
  "period_start_at": "2026-07-06T00:00:00Z",
  "period_end_at": "2026-07-07T00:00:00Z",
  "ready_at": null,
  "claimed_at": null
}
```

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `progress` | Текущее значение прогресса. | Показывать относительно `target_count`. Если больше target, визуально ограничивать target. |
| `status` | Статус задания. | Управляет кнопками. Значения ниже. |
| `period_start_at` | Начало текущего окна выполнения. | Использовать для отладки/таймеров. |
| `period_end_at` | Конец текущего окна выполнения. | Если нужно, показать “обновится через”. |
| `ready_at` | Когда задание стало готовым к claim. | Если есть, можно показать состояние “готово”. |
| `claimed_at` | Когда награда получена. | Если есть, показывать “получено”. |

Статусы:

| Статус | Что означает | Кнопка/реакция |
| --- | --- | --- |
| `open` | Задание еще не выполнено. | Показывать кнопку действия/проверки. Claim выключен. |
| `ready` | Условие выполнено, награду можно забрать. | Показывать активную кнопку claim. |
| `claimed` | Награда уже получена. | Показывать выполненное состояние, claim выключен. |

Если `progress` отсутствует, считать статус `open`, прогресс `0`.

## Награды

```json
{
  "key": "stars",
  "type": "quantity",
  "quantity": 25,
  "scale": 2,
  "unit": null
}
```

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `key` | Ключ предмета/валюты/награды. | Использовать для иконки и локализации через reference/payment catalog. |
| `type` | Тип награды: чаще `quantity`, возможен `duration`. | Для `quantity` показывать число, для `duration` показывать длительность. |
| `quantity` | Количество в minor units. | Отображать с учетом `scale`. |
| `scale` | Количество знаков после запятой. | Например `quantity=25`, `scale=2` отображается как `0.25`; `quantity=100`, `scale=2` как `1.00`. |
| `unit` | Единица длительности, если `type=duration`. | Показывать только для временных наград. |

## Комплексные задания

Комплексное задание имеет `task_kind=complex` и массив `conditions`.

Реакция клиента:

1. Показывать родительское задание как одну карточку.
2. Внутри карточки показать список `conditions`.
3. Для каждого condition использовать те же правила, что для обычного задания.
4. Общий claim показывать только когда родительский `progress.status=ready`.
5. Награда за комплекс находится у родителя в `rewards`.

Поле `conditions[].progress.status` показывает готовность конкретного условия. Родитель становится `ready`, когда все обязательные условия достигли нужного статуса.

## Партнерские задания

Партнерские задания создаются через `task.partnerList`, но возвращаются как обычные `Task`.

Особенности:

| Поле | Как использовать |
| --- | --- |
| `key` | Обычно `partner_issue:<id>`. Передавать как `issue_ref` в `task.partnerCheck` и как ссылку в `task.claim`. |
| `payload` | Публичная информация от партнера: ссылка, кнопка, название задания, тип. Показывать пользователю. |
| `start_mode` | Если `required`, сначала вызвать start и открыть `action_url`. |
| `rewards` | Награда фиксируется в snapshot при выдаче partner issue. Последующее изменение partner reward rules влияет только на новые выдачи. |
| `progress.status` | Управляет кнопками так же, как у обычных заданий. |

Секретные данные партнера никогда не должны попадать в `payload`.

## Публичный `payload`

`payload` специально не стандартизирован жестко. Его согласуют backend и frontend под конкретный тип задания.

Рекомендуемые поля:

| Сценарий | Пример payload | Реакция |
| --- | --- | --- |
| Подписка | `{ "channel": "@MyChannel", "url": "https://t.me/MyChannel" }` | Открыть канал, потом вызвать check. |
| Игра | `{ "game": "pvp", "result": "any" }` | Открыть нужный экран игры. |
| Реклама | `{ "kind": "watch_ads", "required_count": 15 }` | Запустить рекламу и обновлять прогресс. |
| Партнер | `{ "link": "...", "button_text": "Subscribe" }` | Показать кнопку с переходом. |
| Комплекс | `{ "kind": "complex_reward" }` | Использовать как дополнительные данные карточки. |

Нельзя класть в `payload`: токены, webhook secrets, приватные URL с ключами, внутренние payload для проверки.

Для подписок и бустов токены должны лежать в приватном `integration.payload`, а не в публичном `payload`. Пользователю `integration.payload` не отдается в `task.list`; его читает только integration checker во время `task.check`.

Пример приватного `integration.payload` для Telegram-подписки или Telegram-буста:

```json
{
  "platform": "telegram",
  "chat_id": "@MyChannel",
  "token_strategy": "round_robin",
  "tokens": ["<TG_BOT_TOKEN_1>", "<TG_BOT_TOKEN_2>"]
}
```

Пример приватного `integration.payload` для VK-подписки:

```json
{
  "platform": "vk",
  "group_id": "<VK_GROUP_ID>",
  "api_version": "5.199",
  "token_strategy": "round_robin",
  "tokens": ["<VK_ACCESS_TOKEN_1>", "<VK_ACCESS_TOKEN_2>"]
}
```

## Target и видимость

Target применяется сервером до выдачи задания пользователю.

Поддерживаемые критерии:

| Критерий | Как работает |
| --- | --- |
| `is_premium` | Показывать только premium/non-premium пользователям. |
| `sex` | Фильтр по полу. |
| `country` | Фильтр по ключам стран. |
| `loc` | Фильтр по локали. Если пользователь запросил `en`, а target только `ru`, задание не показывается. |
| `platform` | Фильтр по платформе, например `vkma`, `tma`, `ok` или числовой id. |

Фронт не должен повторно фильтровать target. Если задание пришло, его можно показывать.

## Import/export поля

Эти поля используются админкой и переносом конфигурации между workspace.

### Корень пакета

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `format` | Версия формата, сейчас `tasks.export.v1`. | При импорте принимать только поддерживаемый формат. |
| `service` | Должно быть `tasks`. | Защита от импорта пакета другого сервиса. |
| `created_at` | Время экспорта. | Информационное поле. |
| `items` | Метаданные reward items. | Использовать для предпросмотра/документации дампа; каталог предметов живет в reference/payment. |
| `groups` | Группы заданий. | Основная часть импорта. |
| `sequences` | Последовательные цепочки. | Использовать, если задания должны выполняться по порядку. |

### `groups[]`

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `key` | Ключ группы. | Обязателен. |
| `position` | Порядок группы. | Сортировать по нему в админке. |
| `is_active` | Активность группы. | Неактивная группа не должна выдавать задания пользователю. |
| `localization` | Переводы группы. | Показывать в UI. |
| `tasks` | Задания внутри группы. | Импортируются после группы. |
| `partner_configs` | Настройки партнеров для группы. | Требуют секреты при импорте. |
| `partner_reward_rules` | Награды партнерских заданий. | Используются при выдаче partner issues. |

### `tasks[]` в import/export

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `key` | Стабильный ключ задания. | Обязателен. |
| `sequence_key` | Ключ последовательности. | Если указан, должен быть указан `sequence_position`. |
| `sequence_position` | Позиция в последовательности. | Если указана, должен быть указан `sequence_key`. |
| `task_kind` | Тип задания. | См. раздел `task_kind`. |
| `action_key` | Ключ события прогресса. | Бэкенд записывает прогресс по этому ключу. |
| `action_kind` | Класс действия. | См. раздел `action_kind`. |
| `claim_mode` | Режим получения награды. | Обычно `manual`. |
| `start_mode` | Требуется ли start. | `none` или `required`. |
| `target_count` | Цель прогресса. | Для комплексного задания обычно равно количеству условий. |
| `reset.unit` | Единица окна сброса: `never`, `day`, `hour` и т.д. | Определяет периодичность выполнения. |
| `reset.every` | Размер окна. | Например `day/every=1` означает ежедневно. |
| `position` | Позиция в группе. | Сортировка. |
| `payload` | Публичный payload. | Можно показывать пользователю. |
| `target` | Правила таргетинга. | Серверный фильтр отображения. |
| `integration` | Приватная интеграционная настройка. | Не показывать пользователю. |
| `image_url` | Картинка задания. | Показывать, если есть. |
| `is_visible` | Видимость самостоятельной карточки. | Скрытые condition-задания не показывать отдельно, но они могут быть внутри `conditions`. |
| `is_active` | Активность задания. | Неактивные задания не выдаются пользователю. |
| `start_at` | Начало доступности. | До этого времени не показывать. |
| `end_at` | Конец доступности. | После этого времени не показывать. |
| `localization` | Переводы задания. | Использовать для `title`/`description`. |
| `rewards` | Награды задания. | Для conditions может быть пусто; для complex награда обычно на родителе. |
| `conditions` | Условия комплексного задания. | Работает только для `task_kind=complex`. |

### `conditions[]`

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `task_key` | Ключ задания-условия. | Должен существовать в том же import package. |
| `required_status` | Какой статус условия нужен родителю: обычно `ready`, иногда `claimed`. | Если `ready`, достаточно выполнить условие; если `claimed`, надо еще забрать награду condition. |
| `position` | Порядок условия. | Сортировка внутри комплексной карточки. |
| `is_required` | Обязательность условия. | Только обязательные условия участвуют в прогрессе и пользовательском `conditions` родителя. Необязательная задача выполняется самостоятельно и может иметь собственную награду. |

### `partner_configs[]`

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `provider` | Код партнера. | Должен соответствовать зарегистрированному Lua/native provider. |
| `platform` | Платформа настройки. | Используется в `task.partnerList`. |
| `is_enabled` | Включен ли партнер. | Если false, задания не выдаются. |
| `secret` | Секрет для запросов к партнеру. | При экспорте без секретов хранится только ключ, значение передается через `ImportRequest.secrets`. |
| `webhook_secret` | Секрет URL webhook. | Используется в `/webhook/{workspace_id}/{secret}/`. |
| `target` | Target партнерского блока. | Фильтрует выдачу partner tasks. |
| `settings` | Настройки партнера: base URL, action, лимиты и т.д. | Передаются runtime/Lua. |

### `partner_reward_rules[]`

| Поле | Значение | Как реагировать |
| --- | --- | --- |
| `provider` | Код партнера. | Должен совпадать с config. |
| `external_type` | Тип внешнего задания. `*` означает fallback для всех типов. | Выбирает награду под тип задания партнера. |
| `reward` | Награда. | Показывается и выдается как обычная награда. |
| `position` | Порядок правила. | Для админки. |
| `is_enabled` | Включено ли правило. | Если выключено, награда не применяется. |

## Рекомендованная логика UI

1. Получить группы через `task.list`.
2. Для отдельных партнерских секций вызвать `task.partnerList`.
3. Объединять отображение по группам/секциям, но не смешивать разные provider без явной UI-группы.
4. Если `progress.status=claimed`, показывать выполнено.
5. Если `progress.status=ready`, показывать claim.
6. Если `start_mode=required` и задание не started, сначала вызвать start.
7. Если `action_kind=channel_subscribe`, `channel_boost` или `external`, после действия пользователя вызвать check.
8. Если `task_kind=complex`, показывать `conditions` и claim только у родителя.
9. После успешного claim всегда обновлять список или локальную карточку из ответа.
10. Не использовать приватные поля import/export в пользовательском UI.
